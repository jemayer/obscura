# Obscura — Product Requirements Document

## Vision

Obscura is a purpose-built static site generator for photography portfolio websites. It takes a directory of photos organised into galleries, Markdown blog posts, and simple pages, and produces a fully static website with responsive images, a lightbox viewer, and cross-referenced content.

It is not a general-purpose static site generator. Every design decision is made in service of one use case: a photographer publishing their work on the web.

## Target User

A photographer who wants a portfolio site they own and control. They are comfortable editing YAML and Markdown files and running CLI commands, but they are not necessarily a software developer. The workflow should feel closer to "organise your files, run a command, deploy" than to "write code."

## Goals

- Produce fast, fully static sites with responsive images optimised for the web
- Make the common workflow — add photos, write about them, publish — as frictionless as possible
- Automatically extract and surface photo metadata (EXIF) without manual data entry
- Support rich cross-referencing between blog posts and photos
- Ship with a polished default theme suitable for a professional photography portfolio
- Fail loudly on real problems (broken references, unsupported formats) and gracefully on minor ones (missing EXIF fields)

## Non-Goals

- General-purpose static site generation (no plugin system, no arbitrary content types)
- Server-side functionality (no contact forms, no search, no comments)
- Image editing or RAW processing — Obscura works with exported files only
- Internationalisation (planned as a future epic, out of scope for v1)

## Content Model

### Photos

Photos are the primary content type. Each photo belongs to exactly one gallery.

**Directory structure:**

```
site/content/photos/<gallery-slug>/<photo-filename>.{jpg,jpeg,png,tiff,tif,webp}
```

**Metadata** is assembled from two sources, merged at build time:

1. **EXIF data** — read automatically from the image file via exifr
2. **Sidecar YAML** — an optional file with the same name and a `.yaml` extension, placed alongside the image

When both sources provide a value for the same field, the sidecar wins.

**Sidecar auto-generation:** At every build, Obscura generates a sidecar file for any photo that does not yet have one. The sidecar is pre-filled with EXIF data and left for the user to enrich. Existing sidecars are never overwritten.

**Sidecar template:**

```yaml
# Auto-generated by Obscura — feel free to edit
title: ""
date:             # from EXIF if available
camera:           # from EXIF if available
lens:             # from EXIF if available
gps_lat:          # from EXIF if available
gps_lon:          # from EXIF if available
location: ""      # fill in manually
caption: ""
tags: []
```

**Slugs** are derived from the filename, namespaced by gallery: `<gallery-slug>/<filename-without-extension>`. There is no slug field in the sidecar — slugs are a system concern. Duplicate filenames across galleries are handled by the namespace. A shortcode referencing a slug that exists in multiple galleries without specifying the gallery is a hard build error.

### The `post-assets` Gallery

Photos that illustrate blog posts but do not belong to any curated gallery live in `site/content/photos/post-assets/`. This directory behaves identically to any other gallery — sidecars, EXIF reading, image processing, shortcode references (`post-assets/some-photo`), and permalink pages all work the same way. The only difference is that `post-assets` is marked `listed: false` in `site/config/galleries.yaml`, so it does not appear on the gallery index page.

### Galleries

Galleries are declared in `site/config/galleries.yaml`. Adding a new gallery is just adding an entry:

```yaml
galleries:
  - slug: monochromatic
    title: Monochromatic
    description: Black and white photography
    listed: true

  - slug: post-assets
    title: Post Assets
    listed: false
```

The `listed` flag controls whether the gallery appears on the gallery index page. All galleries — listed or not — are fully functional (photo pages, shortcode references, image processing).

### Blog Posts

Blog posts are Markdown files in `site/content/posts/` with YAML frontmatter:

```yaml
---
title: "Berlin in the Rain"
date: 2024-11-15
tags: [street, berlin]
summary: "A walk through Kreuzberg on a rainy November afternoon."
---
```

The `summary` field is optional and used for the RSS feed. If omitted, the feed uses a truncated version of the post body.

**Photo shortcodes:** Blog posts can reference photos using the syntax `{{< photo "gallery-slug/photo-slug" >}}`. This renders as a styled card showing the photo with its metadata (location, camera, date). A post may contain any number of photo references. The relationship between posts and photos is many-to-many.

### Pages

Simple Markdown-driven pages (About, Contact, etc.) live in `site/content/pages/`. Each is a Markdown file with minimal frontmatter:

```yaml
---
title: "About"
---
```

The output path is derived from the filename: `site/content/pages/about.md` produces `/about/`.

## Pages & Routing

| Route | Description |
|---|---|
| `/` | Homepage with "Recent Shots" — latest N photos across all galleries, ordered by date (sidecar `date` overrides EXIF date). N is configurable via `recent_shots_count` in `site/config/site.yaml`. |
| `/photography/` | Gallery index listing all galleries with `listed: true` |
| `/photography/<gallery-slug>/` | Individual gallery page — grid of photos |
| `/photography/<gallery-slug>/<photo-slug>/` | Photo permalink page — full metadata, back-links to referencing blog posts |
| `/blog/` | Blog index — all posts, reverse chronological |
| `/blog/<post-slug>/` | Individual blog post |
| `/about/` | About page (from `site/content/pages/about.md`) |
| `/contact/` | Contact page (from `site/content/pages/contact.md`) |
| `/feed.xml` | RSS feed of blog posts, auto-discovered via `<link rel="alternate">` in every page's `<head>` |
| `/sitemap.xml` | Full sitemap using `base_url` from `site/config/site.yaml` |
| `/404.html` | Custom 404 page, themed, with links to homepage and gallery index |

## Image Pipeline

**Supported input formats:** JPEG, PNG, TIFF, WebP. Any other format (RAW, video, etc.) is a hard build error with a clear message instructing the user to export to a supported format first.

**Processing** is handled by sharp. At build time, Obscura:

1. Reads EXIF data from the original file
2. Generates resized variants at configurable breakpoints (default: `[400, 800, 1200, 2400]`)
3. Converts all variants to WebP at configurable quality (default: 85)
4. Generates thumbnails for gallery grids
5. Produces `srcset` attributes for responsive `<img>` tags

**Output:** Processed images go to `dist/assets/images/`. Originals are never served or copied to `dist/`.

**Version control:** Original image files are gitignored (too large for git). Sidecar YAML files are not ignored and can be committed at the user's discretion.

## Gallery Browsing & Lightbox

Clicking a photo in a gallery grid opens a **PhotoSwipe** lightbox — full-screen, keyboard and swipe navigation, with photo metadata visible. The lightbox includes a "View details" link that navigates to the photo's permalink page.

---

## Theming

Themes are based on CSS custom properties and a theme manifest file. Built-in themes live in `themes/<theme-name>/`; custom themes go in `site/themes/<theme-name>/` (checked first).

Obscura ships with one theme: **`editorial`** — light background, strong typography, inspired by Foam magazine and LensCulture. The `editorial` theme includes a full dark mode variant via `prefers-color-scheme: dark`.

The active theme is set in `site/config/site.yaml`.

## Error Handling

| Condition | Behaviour |
|---|---|
| `{{< photo >}}` shortcode references a non-existent photo | **Hard error** — build fails |
| Ambiguous shortcode slug (matches photos in multiple galleries) | **Hard error** — build fails |
| Unsupported image file format in a gallery directory | **Hard error** — build fails with a message to export to a supported format |
| Missing or corrupt EXIF data | **Warning** — build succeeds. Warning lists each affected file and the specific missing fields |

EXIF warning format:

```
⚠ Missing or corrupt EXIF data in 3 photos:
  - site/content/photos/monochromatic/berlin-rain-2024.jpg (no date)
  - site/content/photos/color/market.jpg (corrupt EXIF)
  - site/content/photos/analog/scan-001.jpg (no camera info)
  Tip: fill in the missing fields in the corresponding sidecar YAML files.
```

## Configuration

### `site/config/site.yaml`

```yaml
base_url: https://coma.photography
title: "coma.photography"
theme: editorial
recent_shots_count: 12
license: all-rights-reserved

# Display fields for photo detail pages and lightbox overlay.
# Options: date, camera, lens, settings, location, tags, license
# "exif" is a shorthand alias for date + camera + lens + settings.
# Defaults to all fields when omitted.
# photo_display_fields: [date, camera, lens, settings, location, tags, license]
# lightbox_display_fields: [date, camera, location, license]

images:
  breakpoints: [400, 800, 1200, 2400]
  webp_quality: 85
```

### `site/config/galleries.yaml`

Declares all galleries with their metadata and listing visibility.

## Developer Workflow

- `npm run build` — full production build to `dist/`
- `npm run dev` — watch mode with rebuild on file changes
- `npm test` — run the full test suite

---

## Testing Strategy

Test coverage is mandatory. A feature is not done until its tests pass.

**Unit tests** cover:

- EXIF reading and fallback behaviour
- Sidecar YAML merging (EXIF + sidecar, sidecar wins on conflict)
- Slug generation and gallery namespacing
- Markdown rendering
- Shortcode parsing and resolution
- Cross-reference graph building (photo-to-post back-links)

**Integration tests** cover:

- Full build pipeline output: correct files at correct paths
- Image variants exist at all configured breakpoints
- Back-links resolve correctly on photo permalink pages
- RSS feed and sitemap contain correct entries
