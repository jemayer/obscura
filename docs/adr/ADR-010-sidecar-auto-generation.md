# ADR-010: Sidecar YAML Auto-Generation Strategy

- **Date:** 2026-02-27
- **Status:** Accepted

## Context

Each photo can have a YAML sidecar file containing metadata (title, location, caption, tags, etc.) that supplements or overrides EXIF data. Creating these files manually for every photo is tedious. But sidecars are also the primary way users enrich their photos with information that EXIF doesn't carry (location names, captions, tags).

We need a strategy that:
- Removes the friction of creating sidecar files from scratch
- Pre-fills what we can from EXIF so the user only fills in what's missing
- Never destroys user edits

## Decision

At every build, comagen checks each photo for a corresponding sidecar YAML file. If none exists, it generates one pre-filled with available EXIF data and empty fields for manual enrichment. Existing sidecars are never overwritten or modified by the build process.

Generated sidecars include a comment header (`# Auto-generated by comagen — feel free to edit`) to signal that the file is meant to be edited.

## Consequences

- Adding a new photo and running a build is enough to get a sidecar — no manual file creation needed
- EXIF data is extracted once and persisted in the sidecar, so users can see and correct it
- Users can commit sidecars to version control, building up a metadata library over time
- The "never overwrite" rule means that if EXIF extraction improves in a future version, existing sidecars won't benefit automatically — users would need to delete and regenerate them. This is an acceptable trade-off for data safety
- Sidecars for photos in `post-assets` work identically to those in any other gallery
